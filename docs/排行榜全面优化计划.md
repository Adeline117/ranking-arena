## 第1阶段：排行榜表格核心增强

### 1.1 优化交易员 API 查询（高复杂度）
- **文件**: `app/api/traders/route.ts`（第 258-338 行）
- **问题**: 每个交易所来源都有 N+1 查询（冷请求时约 60+ 次 Supabase 查询）
- **操作**:
  1. 迁移 00030（唯一约束）执行后，用单一查询 `ORDER BY arena_score DESC` 替换按来源循环
  2. 将 win_rate 归一化（第 373-375 行）移到导入脚本中，使数据库存储已归一化的值
  3. 将非筛选请求的 Redis 缓存 TTL 从 60s 提升到 120s
- **目标**: API 冷启动响应 < 500ms

### 1.2 为所有用户添加 URL 同步的交易所筛选器（中等复杂度）
- **文件**: `app/components/home/RankingSection.tsx`, `app/components/ranking/RankingTable.tsx`, `app/components/premium/AdvancedFilter.tsx`
- **问题**: 交易所筛选功能目前在 Pro 付费墙后面。免费用户也需要基础的交易所筛选。
- **操作**:
  1. 将交易所筛选器从 AdvancedFilter 中提取为独立组件，对所有用户可见
  2. 使用 API 已返回的 `availableSources` 数组填充选项
  3. 在现有 URL 同步逻辑中添加 `exchange` 参数

### 1.3 为大数据集集成 VirtualList（中等复杂度）
- **文件**: `app/components/ui/VirtualList.tsx`（已存在但未使用）, `app/components/ranking/RankingTable.tsx`
- **操作**:
  1. 在分页模式旁添加"虚拟滚动"模式
  2. 表格视图使用 VirtualList，`itemHeight={52}`
  3. 移动端卡片视图使用 `useInfiniteScroll` hook
  4. 保留分页作为免费用户和 SEO 的默认模式

### 1.4 补充缺失的筛选预设（低复杂度）
- **文件**: `app/components/ranking/FilterPresets.tsx`
- **新增预设**: `web3`（DEX 来源）、`consistent`（MDD<30%, 胜率>55%, 分数>50）、`top_performer`（分数>80）

## 第2阶段：交易员详情页优化

### 2.1 修复数据加载架构（中等复杂度）
- **文件**: `app/trader/[handle]/page.tsx`
- **问题**: 完全客户端渲染（无 SSR），useEffect 依赖项错误导致重复请求
- **操作**:
  1. 转为混合模式：使用 `generateMetadata` 服务端布局实现 SEO + 基础资料数据
  2. 从 useEffect 依赖中移除 `showToast` 和 `language`，改用 ref
  3. 添加 SWR 实现客户端 stale-while-revalidate 数据获取

### 2.2 添加评分分解可视化（低复杂度）
- **文件**: `app/components/trader/OverviewPerformanceCard.tsx`, `app/components/ranking/ScoreBreakdownTooltip.tsx`
- 在交易员详情页复用 `ScoreBreakdownTooltip`
- 添加可视化进度条：收益分 (0-70)、盈亏分 (0-15)、回撤分 (0-8)、稳定分 (0-7)

### 2.3 缺失数据占位符与数据来源标注（低复杂度）
- **文件**: `app/components/trader/OverviewPerformanceCard.tsx`, `app/components/trader/TraderHeader.tsx`
- 当 `scoreConfidence` 为 'partial'/'minimal' 时显示置信度徽章
- 对空值指标（win_rate, max_drawdown）显示 "N/A" + 工具提示说明该交易所不提供此指标
- 使用 `profile.source` 添加数据来源徽章

## 第3阶段：数据管道与定时任务优化

### 3.1 实现智能刷新调度（高复杂度）
- **文件**: `vercel.json`, `lib/cron/utils.ts`, `app/api/cron/calculate-tiers/route.ts`
- 迁移 00026 已添加 `activity_tier` + `next_refresh_at` 列
- **操作**:
  1. 在 `calculate-tiers` 定时任务中：根据粉丝数、排名变化、关注列表数量分类
  2. 修改 `fetch-traders/[platform]` 仅刷新 `next_refresh_at <= NOW()` 的交易员
  3. 刷新频率：热门=15分钟、活跃=1小时、普通=4小时、休眠=24小时

### 3.2 在导入脚本中预计算 Arena Score（中等复杂度）
- **文件**: `scripts/import/` 下的 30+ 脚本, `lib/utils/arena-score.ts`
- 创建共享模块 `scripts/import/arena-score-calc.mjs`（ESM 版本）
- 每个脚本在 upsert 时写入 `arena_score`
- API 仅在 `arena_score` 为 null 时回退到 JS 计算

### 3.3 添加数据异常检测（中等复杂度）
- **文件**: `app/api/cron/detect-anomalies/route.ts`
- 标记：快照间 ROI 变化 >500%、MDD 重置、胜率 >95% 且交易数 <10、盈亏跳变 >$1M
- 记录到 `trader_anomalies` 表，按严重程度分级
- 将"严重"异常从排行榜中排除，等待人工审核

## 依赖关系图

```
第0阶段 (数据库 + 稳定化)
   |
   +---> 第1阶段 (排行榜表格) -----> 第4阶段 (性能 + 移动端)
   |         |
   |         +---> 第2阶段 (交易员详情)
   |
   +---> 第3阶段 (数据管道) -----> 第5阶段 (清理 + 测试)
```

第0阶段是所有阶段的前置条件。第1和第3阶段可以并行执行。第2阶段依赖第1阶段。第4阶段依赖第1阶段。第5阶段依赖第3阶段。
