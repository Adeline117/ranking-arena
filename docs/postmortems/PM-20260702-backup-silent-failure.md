# PM-20260702：数据库日备静默失败 3 周（双保险同时失效）

| 字段     | 值                                                                                       |
| -------- | ---------------------------------------------------------------------------------------- |
| 日期     | 2026-07-02 发现；影响窗口 2026-06-12 ~ 07-02（约 20.5 天）                               |
| 严重度   | SEV2（无数据丢失，但 RPO 从 24h 劣化到 20 天——期间任何数据灾难都不可恢复）               |
| 用户影响 | 无直接影响；潜在灾难恢复能力丧失                                                         |
| 检测方式 | 企业级差距整改中新建"备份新鲜度哨兵"首跑即发现（非告警、非人工——此前无任何机制能发现它） |

## 时间线

- 2026-06-12：R2 上最后一份备份 `db-backups/2026/06/arena-backup-20260612.sql.gz`（375.6MB）
- 2026-06-12 ~ 07-02：日备未再运行，无人知晓
- 2026-07-02：`scripts/openclaw/backup-freshness-check.mjs` 首跑报 491.6h > 26h；
  当场手动补跑 `npm run backup:r2`

## 根因

直接原因：本机（Mac Mini）user crontab 为空——备份调度不存在
→ 为什么：约 6-12 前后 crontab 被清空/重建时备份条目丢失（launchd 迁移只迁了
scraper/checkpoint 等，没迁备份）
→ 为什么没发现：唯一兜底 GH Actions `health-monitor.yml` 的 Telegram 告警
因 GH secrets（TELEGRAM_BOT_TOKEN 等）从未配置而**同样静默空转**
→ 系统性根因：**两层保险都是"配置了代码但没配置运行条件"，且都没有
"哨兵挂了本身要告警"的自检**——监控的监控不存在。

## 修复

- 止血：2026-07-02 手动 `npm run backup:r2` 补跑（恢复 RPO）
- 根治：备份新鲜度哨兵入库（>26h 告警 + **哨兵自身失败也告警**，防双死）；
  恢复 crontab 调度（日备 03:30 + 哨兵 09:00，待用户授权写入）；
  GH secrets 配齐 Telegram/Vercel（待用户授权，见差距报告落地记录）

## 教训

调度和凭证是代码之外的"运行条件"，仓库绿 ≠ 生产在跑（与迁移漂移同构：
仓库写了 ≠ 生产有）。任何哨兵必须对"自己没跑成"也发声，否则哨兵之死即盲区。
