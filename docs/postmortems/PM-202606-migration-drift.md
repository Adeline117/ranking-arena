# PM-202606：~200 个迁移从未应用到生产，社交/订阅/支付写路径长期静默 500

| 字段     | 值                                                            |
| -------- | ------------------------------------------------------------- |
| 日期     | 2026-06 审计发现；影响窗口长达数月                            |
| 严重度   | SEV1（发帖/点赞/订阅按钮/支付记录多条写路径断裂，含资金相关） |
| 用户影响 | 用户发帖/点赞/订阅操作静默失败；支付记录缺失                  |
| 检测方式 | 全量审计时发现，非告警——静默 500 被 catch 吞掉，无人报告      |

## 时间线

- 2026 上半年：多个并发 claude 会话各自手工 SQL-editor/MCP 应用迁移、任意命名
  （字母后缀 `20260319h_*` 等无法进 supabase ledger）
- 迁移文件进仓库 ≠ 生产有表；代码凭"用户表应该有 display_name"类先验写查询
- 2026-06：审计发现 ~200 个迁移漂移、多条核心写路径长期 500

## 根因

直接原因：查询引用了生产不存在的表/列 → 为什么：迁移写了没应用/命名无法进 ledger
→ 为什么长期没发现：DB 错误被 catch/safeQuery 静默吞掉
→ 为什么会系统性发生：**多会话并发 + AI 生成速度远超验证速度 + schema 变更无
单一通道**——"仓库写了"和"生产有"之间没有任何强制对账机制。

## 修复

- 止血：逐个核对补齐关键表/列
- 根治：`scripts/new-migration.sh`（纯时间戳命名）+ `supabase db push` 单一通道 +
  `npm run qa:schema`（代码 .from()/.rpc() 依赖 vs 生产实际清单对账）+
  `check-migration-ledger.mjs`（BASELINE=20260611 起可靠核对）+
  每日 schema 金丝雀哨兵（7:30 crontab，失败 Telegram）

## 教训

AI 时代 schema 接地必须是写代码前的反射而非事后追查：写任何 `.from('x')` 前
先确认 x 在生产存在。静默失败让漂移隐形累积。
