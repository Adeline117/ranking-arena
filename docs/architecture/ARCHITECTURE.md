# 系统架构文档

## 概述

Arena 是一个加密货币交易员排行榜与社区平台，聚合多个交易所的跟单数据，提供透明的交易员排名和社区讨论功能。

## 技术栈

| 层级 | 技术 | 说明 |
|------|------|------|
| 前端框架 | Next.js 16 | App Router, Server Components |
| UI 库 | React 19 | 最新的 React 特性 |
| 类型系统 | TypeScript 5 | 严格类型检查 |
| 样式 | Tailwind CSS 4 | 原子化 CSS |
| 状态管理 | Zustand | 轻量级状态管理 |
| 数据库 | Supabase (PostgreSQL) | 托管数据库 + Auth |
| 缓存 | Upstash Redis | 分布式缓存 + 限流 |
| 部署 | Vercel | 边缘部署 + Serverless |
| 监控 | Sentry | 错误追踪 + 性能监控 |

## 数据流

### 1. 交易员数据同步

```
                    Vercel Cron (每6小时)
                            │
                            ▼
┌─────────────────────────────────────────┐
│          /api/cron/fetch-traders        │
│  1. 获取各交易所排行榜                   │
│  2. 数据清洗和标准化                     │
│  3. 存入 trader_snapshots 表            │
└─────────────────────────────────────────┘
                            │
                            ▼
┌─────────────────────────────────────────┐
│              Supabase                   │
│  trader_sources (交易员基本信息)         │
│  trader_snapshots (历史快照)            │
└─────────────────────────────────────────┘
```

### 2. 用户请求流程

```
用户请求 → Middleware (认证/限流) → API Handler → 数据层 → 响应
    │                                    │
    │                                    ├── Supabase (持久化)
    │                                    └── Redis (缓存)
    │
    └── 错误 → Sentry (监控) → Alert (通知)
```

### 3. 实时更新

```
客户端
    │
    ├── Supabase Realtime (WebSocket)
    │   └── 帖子/评论/通知更新
    │
    └── Service Worker (离线缓存)
```

## 安全架构

### 认证流程

```
1. 用户登录 (邮箱 + OTP)
       │
       ▼
2. Supabase Auth 验证
       │
       ▼
3. 返回 JWT Token
       │
       ▼
4. 客户端存储 Token
       │
       ▼
5. 后续请求携带 Token
       │
       ▼
6. Middleware 验证 Token
```

### 安全措施

| 措施 | 实现 | 位置 |
|------|------|------|
| XSS 防护 | DOMPurify 消毒 | `lib/utils/sanitize.ts` |
| CSRF 防护 | 双重提交 Cookie | `middleware.ts` |
| 限流 | Upstash Ratelimit | `lib/utils/rate-limit.ts` |
| CSP | 内容安全策略 | `middleware.ts` |
| 加密 | AES-256-GCM | `lib/exchange/encryption.ts` |
| RLS | 行级安全 | Supabase Policies |

## 性能优化

### 缓存策略

| 数据类型 | 缓存位置 | TTL |
|----------|----------|-----|
| 交易员列表 | Redis + CDN | 60s |
| 用户资料 | Redis | 5m |
| 静态资源 | CDN | 1年 |
| API 响应 | stale-while-revalidate | 30-300s |

### 数据库优化

- 关键字段建立索引
- 使用分页查询
- 批量操作优化
- 连接池管理

## 监控与告警

### 监控指标

- 请求延迟 (P50/P95/P99)
- 错误率
- 数据库连接数
- 内存使用率
- API 调用量

### 告警规则

| 条件 | 级别 | 通知渠道 |
|------|------|----------|
| 错误率 > 5% | Critical | Slack + PagerDuty |
| P99 > 3s | Warning | Slack |
| 数据库连接失败 | Critical | Slack + PagerDuty |
| 限流触发 | Info | Slack |
