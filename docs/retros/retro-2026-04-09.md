## What Could Be Better

1. **49.5% of commits are fixes** — high fix-to-feature ratio (499 `fix`
   vs 75 `feat`). Reasonable for a maintenance-heavy phase, but the gap is
   wide. **Suggested action**: gate `feat` ratio in next week's retro;
   target 15%+ feat commits.

2. **`compute-leaderboard/route.ts` was changed 54 times this week** —
   nearly 8 changes/day on a single file. Today's split (1775 → 940)
   should reduce the surface area, but the file still owns the scoring
   loop, degradation check, and upsert. **Action**: track route.ts
   change-count next week; if still >20/week, do a second-pass split.

3. **7 reverts in 7 days** — 2 are explicit performance reverts:
   - `447c7d668 revert: rollback fetchLimit*2 — caused 20s query timeout`
   - "warm-cache every 5min (was 15min, kept getting reverted)"

   **Action**: add a load-test step before merging perf changes — ideally
   reproduce against a 10K-trader fixture before shipping.

4. **`fix(i18n)` 20 commits/week** — drift across 4 languages. **Action**:
   investigate whether the i18n loader can fail-loud on missing keys
   instead of silently falling back.

5. **`fix(health)` 20 commits/week + `fix(pipeline)` 23 commits/week** —
   the observability layer itself is churning a lot. The health/pipeline
   surface is supposed to be stable enough to *trust*. **Action**: audit
   what's making the health endpoint require 20 fixes/week.

6. **3 tests failing in `siwe/link/route.test.ts`** — from the security
   commit `6c87e1561` (validate domain/uri/chainId). Not from this
   refactor. Needs cleanup by the session that touched it.

7. **No previous retro** — first retro, no trend baseline. Run again next
   week to start the trend lines.
